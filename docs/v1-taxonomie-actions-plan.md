# V1 Taxonomie Des Actions De Plan

## Objectif

Définir une taxonomie simple et stable pour les actions présentes dans les phases de plan.

Cette taxonomie ne classe pas les actions selon leur profondeur psychologique théorique.
Elle les classe selon leur **rôle concret dans l'expérience produit**.

La V1 retenue comporte 4 catégories :

- `habitudes`
- `missions`
- `clarifications`
- `supports`

## Principe Général

Quand on classe une action, la bonne question n'est pas :

- "quel est son sens philosophique ?"
- "quel est son type psychologique profond ?"

La bonne question est :

- "à quoi sert cette action dans le plan ?"
- "qu'est-ce qu'elle demande concrètement à l'utilisateur ?"
- "où est-ce qu'elle l'emmène ?"

## Les 4 Catégories

## 1. Habitudes

### Définition

Action répétée qui doit revenir dans le temps pour créer de la traction, de l'ancrage ou de la régularité.

### Test simple

Si l'action doit être répétée régulièrement, c'est une habitude.

### Exemples

- rituel du soir
- 10 minutes de marche chaque matin
- respiration quotidienne
- noter son niveau d'énergie chaque soir
- routine d'ouverture de journée

### Ce que les habitudes font

- ancrent
- stabilisent
- construisent une base
- donnent de la continuité

### Ce que les habitudes ne sont pas

- une action ponctuelle
- une redirection vers un outil
- une réflexion de repérage

## 2. Missions

### Définition

Action concrète d'avancée dans le réel.

La mission produit un mouvement observable dans le monde, dans la relation, dans le travail, dans la santé, dans l'environnement ou dans une décision.

### Test simple

Si l'action fait avancer concrètement quelque chose dans le monde réel, c'est une mission.

### Exemples

- envoyer un message
- appeler quelqu'un
- jeter les cigarettes
- prendre un rendez-vous
- préparer un entretien
- faire une séance
- publier une offre
- ranger un espace déclencheur

### Ce que les missions font

- débloquent
- déplacent
- confrontent au réel
- produisent des preuves

### Ce que les missions ne sont pas

- une habitude répétée
- un objet de clarification
- une simple ouverture de module

## 3. Clarifications

### Définition

Action qui sert à se repérer dans la route, relire une situation, comprendre où l'on en est, voir les options et choisir la bonne direction.

La clarification ne sert pas d'abord à soutenir le moteur de transformation.
Elle sert à mieux lire la route et à mieux tourner le volant.

### Test simple

Si l'action aide surtout à se repérer, s'orienter, relire, décider ou comprendre la route, c'est une clarification.

### Exemples

- clarifier où on en est dans la phase
- distinguer le vrai blocage du faux blocage
- relire une situation confuse
- choisir entre deux directions
- comprendre ce qui a réellement déraillé
- voir si l'on doit continuer A ou passer à B

### Ce que les clarifications font

- orientent
- éclairent
- aident à décider
- réduisent le brouillard
- évitent de tourner au hasard

### Ce que les clarifications ne sont pas

- des cartes de défense
- des frameworks de labo
- des protocoles
- des redirections vers un outil du laboratoire ou de l'atelier

Important :

Même si une clarification peut avoir une dimension cognitive ou stratégique, dès que l'action consiste à aller travailler le moteur via le laboratoire ou l'atelier, elle bascule côté `supports`.

## 4. Supports

### Définition

Action qui redirige vers une surface ou un outil d'aide pour soutenir le moteur de la transformation.

Les supports sont la catégorie qui relie le plan aux modules du produit, notamment :

- le laboratoire
- l'atelier d'inspiration
- les outils guidés
- les cartes
- les frameworks
- les protocoles

### Test simple

Si l'action envoie vers le laboratoire, l'atelier d'inspiration ou un outil d'aide qui soutient la transformation, c'est un support.

### Exemples

- aller valider une carte de défense
- revoir une carte d'attaque déjà préparée
- ouvrir un framework dans le laboratoire
- compléter un protocole guidé
- consulter un support spécifique dans l'atelier d'inspiration
- valider une hypothèse préremplie dans un outil
- lancer l'action `défenseur` d'une carte de défense depuis le plan
- lancer l'action principale d'une carte d'attaque depuis le plan
- lancer le geste de soutien d'une carte de soutien depuis le plan

### Ce que les supports font

- soutiennent le moteur
- donnent du carburant
- donnent des leviers
- donnent des objets d'aide activables
- servent de pont dynamique vers les modules
- rendent visibles dans le plan des moyens concrets d'accomplir la phase

### Ce que les supports ne sont pas

- une mission réelle dans le monde
- une habitude répétée
- une clarification de route

## Exercices Et Actions Dérivées Des Cartes

Point produit important :

les exercices issus du labo ne doivent pas rester cachés dans le labo.

Dans beaucoup de cas, ce sont eux qui permettent réellement d'accomplir le plan.

Exemples :

- l'action `défenseur` d'une carte de défense
- l'action centrale d'une carte d'attaque
- le geste principal d'une carte de soutien

Ces éléments doivent pouvoir remonter dans le plan sous forme d'actions visibles et cliquables.

## Règle Recommandée

Quand une carte contient une action particulièrement utile pour la phase :

- cette action peut être matérialisée dans le plan
- elle reste liée à sa carte d'origine
- elle s'ouvre de manière cliquable vers le bon objet du labo

Autrement dit :

- l'objet source reste dans le labo
- mais l'action utile apparaît aussi dans le plan

## Pourquoi C'Est Important

Cela permet :

- de mettre en avant les bons gestes
- de ne pas laisser les cartes enfermées dans une sous-surface
- de faire apparaître les vrais moyens d'exécution dans le plan lui-même
- de montrer que les exercices ne sont pas "à côté" du plan, mais au service du plan

## Ce Que Devient Un Exercice Dans Cette Logique

Un exercice n'a pas besoin d'être une cinquième catégorie.

Il peut exister comme :

- une action dérivée d'un `support`
- visible dans le plan
- cliquable
- liée à la carte ou au support qui la porte

## Forme Produit Recommandée

Le plan peut afficher ce type d'item comme :

- un `support` actif
- avec un CTA direct
- ouvrant la bonne carte au bon endroit

Exemples :

- `Faire ton geste défenseur quand l'envie monte`
- `Lancer ton rituel d'attaque du matin`
- `Revenir à ton geste de soutien du soir`

## Règle De Classification

Même si l'utilisateur voit une action concrète dans le plan, si cette action vient d'une carte, d'un protocole ou d'un outil du labo :

- elle reste classée en `supports`

Ce n'est donc pas :

- une mission
- ni une habitude par défaut

C'est :

- un `support` rendu actionnable dans le plan

## Distinction Clé : Clarification vs Support

Cette frontière est la plus importante de la taxonomie.

## Une clarification

sert à :

- se repérer
- mieux lire la situation
- mieux choisir la direction
- comprendre où l'on en est

Image mentale :

- route
- volant
- orientation
- embranchement

## Un support

sert à :

- aller dans le laboratoire
- aller dans l'atelier
- utiliser un outil d'aide
- activer une carte, un framework, un protocole
- soutenir le moteur de transformation

Image mentale :

- moteur
- carburant
- système d'aide
- module d'assistance

## Règle Produit Fondamentale

Même si un framework aide à réfléchir, s'il est porté par le laboratoire ou l'atelier et sert le moteur de transformation, il doit être classé en `supports`.

La clarification ne doit pas devenir un fourre-tout pour tous les objets "intelligents".

## Règle De Tri Rapide

### Si l'action est répétée

=> `habitudes`

### Si l'action fait avancer quelque chose dans le réel

=> `missions`

### Si l'action sert surtout à se repérer, comprendre la route, choisir la direction

=> `clarifications`

### Si l'action redirige vers le laboratoire, l'atelier ou un outil d'aide

=> `supports`

## Exemples De Classement

### Exemple 1

"Faire 10 minutes de respiration chaque soir"

=> `habitudes`

### Exemple 2

"Envoyer le message à ton manager"

=> `missions`

### Exemple 3

"Clarifier si ce qui te bloque est de la peur, de la fatigue ou du flou"

=> `clarifications`

### Exemple 4

"Aller valider les cartes de défense préparées pour tes moments de craving"

=> `supports`

### Exemple 5

"Ouvrir le framework sur les conversations difficiles dans l'atelier d'inspiration"

=> `supports`

### Exemple 6

"Relire ce qui a réellement déraillé dans ta semaine avant de choisir la prochaine direction"

=> `clarifications`

### Exemple 7

"Compléter le protocole de préparation avant entretien"

=> `supports`

### Exemple 8

"Appeler la salle de sport pour réserver ton premier créneau"

=> `missions`

### Exemple 9

"Faire le geste défenseur prévu pour le moment où tu rentres stressé"

=> `supports`

### Exemple 10

"Lancer ton texte d'attaque du matin"

=> `supports`

## Lien Avec Le Laboratoire Et L'Atelier

Dans cette taxonomie :

- le laboratoire et l'atelier ne sont pas des catégories
- ce sont des surfaces cibles

Le fait qu'une action redirige vers ces surfaces la fait entrer dans `supports`.

Donc :

- `supports` est la catégorie passerelle entre le plan et les modules dynamiques du produit
- `supports` peut aussi faire remonter dans le plan les actions utiles portées par ces modules

## Pourquoi Cette Taxonomie Est Utile

Elle permet de :

- garder le plan lisible
- éviter les catégories instables
- faire de `supports` la vraie catégorie d'orchestration dynamique
- préserver `clarifications` comme catégorie de repérage et d'orientation
- éviter de mélanger route et moteur
- faire exister les exercices du labo directement dans le plan sans créer une nouvelle catégorie

## Version Finale V1

La structure recommandée des phases de plan est donc :

- `habitudes`
- `missions`
- `clarifications`
- `supports`

Avec la logique suivante :

- `habitudes` = ce qui revient
- `missions` = ce qui fait avancer dans le réel
- `clarifications` = ce qui aide à se repérer dans la route
- `supports` = ce qui redirige vers les modules et soutient le moteur
